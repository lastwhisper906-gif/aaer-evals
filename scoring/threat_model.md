# Threat Model — v1 (Phase 2-2): 피평가자 입력의 누출·오염 위협 모델

> Authored by Claude Code, pending human audit (GA-001 (b), D15).
> 대상: 피평가자에게 도달하는 모든 정보 표면. freeze 커밋으로 고정.

## 1. 필드 인벤토리 (evaluatee_input v1.1 + 파이프라인 부가 페이로드)

| 필드 | 타입 | 예시 | 재식별/누출 위험 주석 |
|---|---|---|---|
| case_id | string `case_NN` | `case_01` | **낮음** — 중립 ID(OV-001). 위험: 셔플 시드 노출 시 순서 상관 역산 (시드는 채점 쪽 보관) |
| ticker | string | `MON` | **회사 식별 — 의도된 노출**. 위험은 재식별이 아니라 **결과 상기**(L-1): 회사 식별 = 모델 내부 지식의 열쇠 |
| cik | string | `0001110783` | ticker와 동일 축 (회사 식별 앵커 — 파이프라인 수집용 필수) |
| company_name | string | `Monsanto Company` | 동일 축 + **시제 위험**(OV-002 방어 완료: 컷오프 시점 사명만) |
| cutoff_date | date | `2011-06-28` | **중간** — ① 쌍 날짜 공유로 그룹 힌트(GP-9, 독립 호출로 완화) ② 날짜 자체가 "왜 이 날?" 질문 유발 — 폭로일 전일임을 모델이 추정 가능(잔여 위험, 제거 불가 — 한계 기록) |
| 재무 시계열 (부가) | 구조화 표 | XBRL 개념표 | 값 자체가 회사 지문(fingerprint) — 교란 후에도 비율 구조로 재식별 가능 (D8 소유자 인지: 교란은 측정 장치이지 제거 장치 아님) |
| 제출물 연대기 (부가) | form/date 목록 | `NT 10-K@2018-03-02` | NT·8-K 패턴이 T2 신호 (의도된 채널) — 위험: 특징적 제출 패턴 역시 회사 지문 |

## 2. 공격 벡터 목록 → 4중 방어 매핑

| # | 벡터 | 방어 | 커버 |
|---|---|---|---|
| V1 | 정답 직독: group/scheme 필드 혼입 | 방어 ①(재생성 대조) + ②(additionalProperties:false) + ③(금지 필드 목록) | 완전 |
| V2 | 값 수준 인코딩: ID 접두사·후신 사명 (OV-001/002 재발) | 방어 ④(값 수준 스캔: T/C 접두사, n/k/a 부분열) + 신설: 베이스라인 마커 스캔 | 완전 (알려진 패턴에 한함 — 신종 값 인코딩은 인간 표본 점검 몫) |
| V3 | 쌍 컷오프 날짜 → 기저율 힌트 | GP-9 ① + 케이스별 독립 호출 (GP-6 3조항) | 실행 형태로 완화 |
| V4 | 저장소 컨텍스트 주입 (CLAUDE.md·채점 문서가 시스템 프롬프트에 실림) | 페이로드 규약: 피평가자 호출은 **직접 API 호출**로만 — 세션/서브에이전트 경유 금지 (decisions_log 모델 고정 기록에 명문화) | 실행 규약 |
| V5 | 모델 내부 지식 (학습 데이터 암기) | **차단 불가** — 측정만 가능: 인지 프로브(D7)·교란 delta(D8)·축어 프로브·memorization_suspect(L-1) | 측정·공개 |
| V6 | 베이스라인 출력 혼입 (1-1) | 누출 테스트 (`test_baseline_outputs_not_in_evaluatee_payload`) | 완전 (CI) |
| V7 | 채점자→피평가자 역류 (grader 자료가 후속 피평가자 호출에 잔존) | 케이스별 독립 호출 + 피평가자/채점자 호출 코드 분리 (러너 구현 시 상호 import 금지 테스트) | Phase 3 구현 시 테스트 추가 |
| V8 | 미래 학습 유입 (본 저장소가 차기 모델 학습에 포함 → 차기 사이클 오염) | 카나리 GUID (D9) — 정답 키·내부 문서에 삽입, 차기 사이클에 GUID 상기 여부 프로브 | 장기 감지 |
| V9 | 프롬프트 문언의 분류 체계 힌트 (D4 중립화 실패) | eval spec §1 중립화 + self-red-team 점검 항목 | 문서·검토 |
| V10 | 교란 변형의 불완전성: 이름은 익명인데 제출물 연대기·비율 지문으로 재식별 | 제거 불가 (D8 소유자 결정: 날짜 이동 없음) — 결과 문서에 "교란은 암기 기여의 **측정**" 명문 의무 | 명시적 잔여 위험 |

## 3. 미커버 벡터의 구현 (신설분)

- **V7 구현 예정**: Phase 3 러너에 채점자/피평가자 모듈 물리 분리 + 정적 스캔
  (피평가자 러너가 scoring/를 import하면 실패).
- **V4 구현**: 러너는 순수 `anthropic` SDK 직접 호출, 시스템 프롬프트는 eval spec
  §1 문언만. 러너 코드에 저장소 문서 경로 참조가 없음을 정적 스캔.
- **V8 구현**: `scoring/canary.md`에 GUID 2개 (정답 키용·내부 문서용) — Phase 2-3.

## 4. 잔여 위험 종합 (제거 불가 — 측정·공개 대상)

1. **모델 내부 지식** (V5): 세 채널 중 유일하게 차단 불가(L-1). 인지 프로브 임계
   ≥3/8 (D7, 사전 등록) 분기로 보고 방식이 바뀔 뿐, 위험 자체는 상수.
2. **컷오프 날짜의 의미 추정** (cutoff_date가 폭로 전일이라는 설계 규칙을 모델이
   학습 데이터의 사건 연표와 결합하면 그룹 추정 가능) — 완화: 대조군도 동일 규칙
   (복사)이라 날짜만으로는 무구별. 잔여: 실험군 회사의 유명 사건 일자와 일치.
3. **교란 후 재식별** (V10): 내용 기반 재식별은 원리적으로 잔존 (D8).

**학습 노트(§10)**: 방어의 강도는 "몇 겹인가"가 아니라 "어느 채널을 못 보는가"로
평가한다 — 4중 방어는 전부 **데이터 표면**을 보고, V5(모델 내부)와 V4(실행 표면)는
데이터 밖에 있다. 그래서 실행 규약과 프로브가 방어의 나머지 절반이다.
